# 第 1 周：学生管理系统 v1

## 本周目标

练习变量、基本类型、流程控制、数组、方法、类和对象，使用固定数组实现第一个学生管理程序。

## 已实现功能

- 新增学生并检查重复学号
- 查看学生列表
- 按学号查询
- 修改学生
- 删除学生并移动后续数组元素
- 校验学号、姓名、年龄、性别和手机号
- 处理菜单和年龄的非数字输入

## AI 代码审查记录

本周保留 v1 原始实现作为学习基线，审查发现和后续重构如下：

| 审查发现 | 风险或限制 | 后续处理 |
| --- | --- | --- |
| `Student[10]` 固定容量 | 最多保存 10 人，还要手动维护 size | v2 改为 `ArrayList<Student>` |
| 所有菜单逻辑都在 main | 方法过长，新增功能容易影响其他分支 | v2 拆分新增、查询、修改、删除方法 |
| 变量存在 `In`、`Id`、`id_tag` 等混合命名 | 不符合 Java 驼峰习惯，阅读成本高 | v2 统一为 scanner、id、found 等名称 |
| App 直接遍历和修改数组 | 输入输出与存储耦合 | v2 引入 Repository 接口 |
| 学号可以通过 setter 修改 | 作为唯一标识时可能破坏集合索引关系 | v2 将学号改为 final |
| 性别使用任意 String | 依赖字符串拼写 | v3 改为 Gender 枚举 |
| 没有自动化测试 | 修改后只能手动运行验证 | 最终 Maven 版加入 JUnit |

## 本周主要收获

- 数组长度固定，有效元素数量要单独记录。
- 删除数组元素后，需要把后续元素向前移动，并清空最后一个有效位置。
- 类把学生的多个属性组合成一个对象。
- 构造方法和 setter 可以集中维护数据合法性。
- `switch` 适合处理固定菜单选项，循环负责让程序持续运行。

v1 不继续重构，是为了保留第一周真实学习结果。审查意见已经在 v2、v3 和最终项目中逐步落实，因此可以直接比较“原始实现”和“重构实现”的差异。
